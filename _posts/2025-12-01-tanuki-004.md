## Description

This is a beginner Broken Access Control (BAC) lab on [Bugforge.io](https://bugforge.io) that abuses a type of Insecure Direct Object Reference (IDOR) known as a Broken Object Level Authorization (BOLA) vulnerability to read data belonging to another user.

## Attack Path

As we review the application, notice that there is a functionality that allows us to view our flashcard review stats.

![](assets/img/2025-12-01-tanuki-004/file-20251201094907955.png){: .shadow .rounded-corners }

Notice that the request directly references the user ID 4 in the URL. What if we modified the request slightly to request the data for another user? The following request and response illustrate that:

![](assets/img/2025-12-01-tanuki-004/file-20251201095039197.png){: .shadow .rounded-corners }
_Modifying the user ID reference to retrieve the admin user's stats_

When we change the user ID from 4 to 1, we get the stats for the admin user, as well as the flag.

## Impact

When objects are directly referenced (such as a user ID) and access controls are not properly enforced, an attacker may be able to view data belonging to another user. The actual impact is heavily dependent on the application and what information is disclosed. In this case the impact is relatively low because the information disclosed is not overly sensitive; however, if an attacker were able to access personal information or credentials, the impact would be significantly higher. 

## Recommendation

Ensure that access controls are strictly enforced to prevent unauthorized access to data. 

## Key Takeaway

Always fully test each application functionality and keep an eye out for direct references to objects like user IDs. Make sure to test any such requests to see if it’s possible to view data belonging to another user.
